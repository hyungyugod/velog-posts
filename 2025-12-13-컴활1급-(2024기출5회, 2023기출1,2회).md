# 📌 1. 누적 세로 막대형 차트의 ‘레이블에 값이 표시’ 의미

## 1-1. 막대(데이터)와 레이블(데이터 레이블)의 구분
- **막대(세로 막대, Column)**: 데이터 크기를 **길이/높이**로 표현하는 시각 요소이다.
- **레이블(Data Label)**: 막대(또는 누적된 조각)에 **텍스트로 붙는 설명**이다.
- 결론: **레이블은 막대를 의미하지 않는다.** 막대는 데이터, 레이블은 그 데이터의 ‘설명 글자’이다.

## 1-2. “레이블 내용으로 값이 표시”의 정확한 뜻
- 차트에서 **데이터 레이블을 켜고**, 레이블에 표시할 항목 중 **‘값(Value)’**을 선택한 상태를 말한다.
- 즉 막대(또는 누적 조각) 위/안에 **숫자 값이 직접 적혀 보이는 상태**이다.

## 1-3. 누적 세로 막대형에서 레이블이 가리키는 값의 두 경우
- **각 누적 조각의 개별 값 표시**: 누적된 각 색상 조각마다 해당 값이 표시됨.
- **누적 합계(총합) 표시**: 막대 꼭대기(또는 특정 위치)에 합계만 표시됨.
- 시험/문장 해석 포인트: “레이블 **내용**”은 레이블에 무엇을 표시하는지(값/이름/백분율/셀 값 등)를 구분시키는 표현이다.

## 1-4. 설계 철학적 관점
- 차트는 원래 **길이 비교(시각적 비교)**를 위한 도구이다.
- 레이블은 여기에 **정확한 수치(읽을 수 있는 정보)**를 더해, “눈대중”을 “읽기”로 바꾸는 보조 장치이다.
- 그래서 “레이블 내용 = 값”은 차트를 **정밀 해석 가능**하게 만드는 선택지로 설계되었다.

# 📌 2. 데이터 통합(Consolidate)에서 ‘첫 행/첫 열 레이블’의 의미

## 2-1. “첫 행/첫 열을 레이블로 사용”의 뜻
- 데이터 통합에서 원본 범위의
  - **첫 행 = 열 레이블**
  - **첫 열 = 행 레이블**
  로 간주하여 **항목 이름으로 매칭**해 통합한다.
- 핵심: **순서(위치)가 아니라 레이블(이름)**로 데이터를 맞춘다.

## 2-2. “원본 데이터에 표시된 순서와 상관없이 통합”이 가능한 이유
- 원본 시트들에서 항목 배치가 다를 수 있다.
- 레이블 기준 통합을 쓰면 엑셀은
  - 같은 이름의 행/열을 찾아
  - 그 값을 같은 항목으로 묶어
  - 합계/평균 등 지정한 함수로 통합한다.
- 따라서 시트 A가 `사과, 배, 포도` 순서이고 시트 B가 `포도, 사과, 배` 순서여도 레이블이 같으면 정상 통합된다.

## 2-3. 설계 철학적 관점
- 통합은 현실 데이터(사람이 만든 표)가 **정렬이 제각각**인 상황을 전제로 한다.
- 그래서 “위치 기반”과 별도로 “의미(레이블) 기반”을 제공하여 **데이터 의미 정렬**을 가능하게 했다.

# 📌 3. ‘원본 데이터에 연결’에서 원본/대상 범위가 같은 시트면 불가한 이유

## 3-1. ‘원본 데이터에 연결’이란?
- 통합 결과가 단순 값이 아니라 **원본을 참조**하여
- 원본이 바뀌면 통합 결과도 **자동 갱신**되도록 하는 동적 연결 옵션이다.

## 3-2. 같은 시트에서 연결을 막는 핵심 이유: 순환 참조 위험
- 원본과 대상이 같은 시트면
  - 대상(통합 결과)이 원본을 참조하는데,
  - 그 원본 범위가 대상 셀을 포함하거나 겹칠 가능성이 커진다.
- 그러면 계산 구조가 $A1$이 $A1$을 필요로 하는 형태(자기참조)가 되어 **순환 참조(Circular Reference)**가 발생할 수 있다.
- 엑셀은 이런 구조를 기본적으로 피하려고 하므로, 같은 시트에서는 연결 생성이 제한된다.

## 3-3. 설계 철학적 관점
- 연결 기능은 편의성이 크지만, 계산 안정성을 해칠 수 있다.
- 그래서 엑셀은 “편의(자동 갱신)”보다 “안전(순환 참조 방지)”을 우선하는 제한을 둔다.

# 📌 4. 사용자 지정 연속 데이터 채우기와 사용자 지정 목록의 제약

## 4-1. “사용자 지정 목록에는 텍스트 또는 텍스트/숫자 조합만 포함”의 의미
- 사용자 지정 목록(Custom List)은
  - 엑셀이 **계산해서 다음 값을 만드는 기능**이 아니라
  - **미리 정의된 ‘목록’을 순서대로 꺼내 쓰는 기능**이다.
- 그래서 목록 항목은 기본적으로 **문자열(텍스트)** 성격이어야 하며,
  - 텍스트만: `상, 중, 하`
  - 텍스트/숫자 조합: `1학년, 2학년, 3학년` / `Part1, Part2`
  처럼 “문자열 패턴”이 된다.

## 4-2. 왜 ‘순수 숫자만’ 목록은 제한되는가?
- `1, 2, 3` 같은 순수 숫자 연속은 엑셀이 이미
  - 자동 채우기(수학적 규칙)로 처리할 수 있다.
- 사용자 지정 목록에 숫자만 허용하면
  - 계산 연속인지, 목록 연속인지 경계가 흐려져
  - 동작 일관성이 깨질 수 있다.
- 따라서 사용자 지정 목록은 **의미 기반(문자열 기반) 연속**에 집중하도록 설계된다.

## 4-3. 설계 철학적 관점
- 엑셀의 연속 채우기는 크게 두 세계로 나뉜다:
  - **계산 엔진(숫자/날짜)**: 규칙을 계산해 생성
  - **목록 사전(사용자 지정 목록)**: 미리 정한 순서를 재생
- 사용자 지정 목록의 “텍스트 중심”은 이 역할 분담을 명확히 하려는 설계다.

# 📌 5. 입력 마스크 기호: L, &, A, 0 (필수)와 반대되는 선택 기호

## 5-1. 필수(반드시 입력) 기호 4개
- **L**: 필수 문자(알파벳). 숫자/공백 불가.
- **&**: 필수 모든 문자(무엇이든 가능: 문자/숫자/특수/공백). 단, 반드시 입력.
- **A**: 필수 문자 또는 숫자. 특수/공백 불가.
- **0**: 필수 숫자. 공백 불가.

## 5-2. 반대되는 ‘선택(입력해도 되고 안 해도 됨)’ 기호
- **L ↔ ?**
  - **?**: 선택 문자(알파벳). 비워도 됨.
- **& ↔ C**
  - **C**: 선택 모든 문자(무엇이든). 비워도 됨.
- **A ↔ a**
  - **a**: 선택 문자 또는 숫자. 비워도 됨.
- **0 ↔ 9**
  - **9**: 선택 숫자. 비워도 됨.

## 5-3. 핵심 암기 구조(대칭)
- 큰 틀: **대문자/강한 기호 = 필수**, 소문자/대응 기호 = 선택
- 시험 함정: `0`(필수 숫자)와 `9`(선택 숫자)를 뒤집어 외우지 않기.

## 5-4. 설계 철학적 관점
- 입력 마스크는 데이터 입력 단계에서 **형식을 강제**하여 데이터 품질을 올린다.
- “필수/선택”을 기호 한 글자에 넣어 **간결한 규칙 언어(미니 DSL)**로 만든 것이 핵심이다.

# 📌 6. Access 보고서: 그룹화/정렬의 ‘같은 페이지에 표시 안함’과 그룹 머리글 위치

## 6-1. ‘같은 페이지에 표시 안함’의 의미
- 그룹화/정렬 옵션에서 **그룹 머리글이 페이지 하단에 혼자 남고**, 상세 데이터가 다음 페이지로 넘어가는 상황을 막기 위한 옵션이다.
- 동작 논리:
  - 현재 페이지에 **그룹 머리글 + 해당 그룹의 일부 상세 데이터**를 함께 배치할 공간이 없으면,
  - 그룹 머리글을 **다음 페이지로 통째로 이동**시켜
  - 머리글과 데이터가 **같은 페이지에서 시작**하도록 만든다.
- 핵심: “머리글이 고아처럼 분리되는 것” 방지.

## 6-2. 제품코드에 대한 그룹 머리글은 어디에 ‘붙는가’
- 제품코드 기준으로 그룹화하면,
  - **제품코드 값이 바뀌는 지점(새 그룹 시작)** 바로 위에 그룹 머리글이 출력된다.
- 출력 흐름 예:
  - (제품코드 = P001) 그룹 머리글
  - P001의 상세 레코드들
  - (제품코드 = P002) 그룹 머리글
  - P002의 상세 레코드들
  - … 반복

## 6-3. “그룹 머리글”은 속성명인가?
- 아니다.
- “그룹 머리글”은 컨트롤의 속성이 아니라, 보고서가 가지는 **구역(Section)**이다.
- 그룹화를 설정하면 Access가 `GroupHeader0`, `제품코드 머리글` 같은 형태로 **머리글 구역을 자동 생성**한다.

## 6-4. 설계 철학적 관점
- Access 보고서는 데이터를 “표”가 아니라 **문서(리포트)**로 취급한다.
- 그룹 머리글/바닥글은 “문서의 장(章) 경계”처럼 작동하며,
- ‘같은 페이지에 표시 안함’은 문서 가독성을 위한 **레이아웃 규칙**에 해당한다.
