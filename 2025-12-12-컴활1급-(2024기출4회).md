# 📌 1. 목록상자(List Box)와 콤보상자(Combo Box)

## 1-1. 공통점
- 둘 다 “미리 정해진 값 중에서 선택”하게 하여 입력 실수를 줄이고 데이터 일관성을 높이는 컨트롤이다.
- 보통 테이블/쿼리의 값 목록을 가져와 선택하도록 만든다.

## 1-2. 핵심 차이
- 목록상자(List Box)
  - 항목 목록이 화면에 펼쳐져 보이는 형태(여러 줄이 항상 보임).
  - 선택 중심(typing보다는 클릭/선택).
  - 다중 선택이 가능한 경우가 많아 “여러 개를 동시에 고르는” 상황에 적합하다.
- 콤보상자(Combo Box)
  - 기본은 한 줄로 접혀 있고 클릭하면 드롭다운으로 펼쳐짐.
  - 선택 + 직접 입력(입력 허용 여부는 속성으로 제한 가능).
  - 값이 많아 화면 공간을 줄여야 하거나, 검색/입력이 섞이는 상황에 적합하다.

## 1-3. 설계 관점(철학)
- 목록상자: 통제(허용된 값만 고르기)와 집합 선택에 강함.
- 콤보상자: 편의(공간 절약, 입력/선택 혼합)와 유연성에 강함.


# 📌 2. 모달 속성에서 “사용자 지정 대화상자” 의미

## 2-1. 대화상자(dialog)의 핵심
- 작업 흐름을 잠시 멈추고, 사용자에게 결정/입력/확인을 요구하는 창이다.

## 2-2. 사용자 지정 대화상자란
- 엑세스에서 개발자가 직접 만든 “폼(Form)”을 대화상자처럼 쓰는 방식이다.
- 흔히 다음 성격을 가진 폼을 의미한다.
  - Modal = Yes (모달)
  - Pop Up = Yes (팝업)
  - 필요 시 Border Style = Dialog
  - 필요 시 Record Selectors/Navigation Buttons 숨김

## 2-3. 모달(Modal)인 이유
- 모달 폼이 열리면 뒤의 다른 폼/리포트 조작이 불가능하다.
- 즉 “지금 이 결정을 끝내야 다음으로 간다”를 강제한다.

## 2-4. MsgBox와 차이
- MsgBox: 단순 확인/경고/선택(버튼과 문구가 제한됨).
- 사용자 지정 대화상자: 입력 필드(텍스트/콤보/체크), 설명 UI, 업무 맥락에 맞춘 화면을 자유롭게 구성 가능.


# 📌 3. 엑세스의 IIf 함수(‘llf’처럼 보이는 것)

## 3-1. 정체: IIf
- “llf”라고 보인 것은 대개 IIf(대문자 I 두 개)가 소문자 l처럼 보인 경우다.

## 3-2. 기본 구조
- IIf(조건식, 참일 때 값, 거짓일 때 값)
- 예: IIf([점수] >= 60, "합격", "불합격")

## 3-3. 성격(If문과 차이)
- IIf는 “흐름 제어”가 아니라 “값 반환”을 위한 표현식 함수다.
- 쿼리의 계산 필드나 컨트롤 원본처럼 “값 하나를 만들어야 하는 자리”에서 자주 사용된다.

## 3-4. 주의 포인트(시험 함정)
- IIf는 상황에 따라 참/거짓 인수 중 필요 없는 쪽도 평가되어 오류가 날 수 있다(예: 0으로 나눔, Null 처리 미흡 등).
- 그래서 Nz 같은 Null 처리 함수와 함께 쓰거나 로직을 분리하는 설계를 고려한다.


# 📌 4. 엑셀 통합문서 보호: 구조 vs 창

## 4-1. 구조(Structure) 보호
- 보호 대상: “시트의 존재/관계(뼈대)”
- 제한되는 작업 예:
  - 시트 추가/삭제
  - 시트 이름 변경
  - 시트 이동/복사
  - 시트 숨기기/숨김 해제
- 의미: “이 통합문서에 어떤 시트들이 있어야 하는지”를 고정한다.

## 4-2. 창(Windows) 보호
- 보호 대상: “통합문서가 화면에 뜨는 방식(프레임)”
- 제한되는 작업 예:
  - 창 크기/위치 변경
  - 최소화/최대화
  - 같은 통합문서를 여러 창으로 띄우는 동작 등
- 의미: 레이아웃이 깨지지 않게 ‘화면 상태’를 고정한다.

## 4-3. 핵심 요약
- 구조: 논리적 구조(시트 구성) 보호
- 창: 시각적/화면 상태 보호


# 📌 5. 모든 시트 선택 vs 코드 보기 vs 시트 보호 작업(엑셀)

## 5-1. 모든 시트 선택(Select All Sheets)
- 의미: “적용 범위”를 통합문서 전체 시트로 확장한다.
- 효과: 서식/행열 너비/페이지 설정 같은 작업이 모든 시트에 동시에 반영될 수 있다.
- 주의: 의도치 않게 모든 시트가 같이 바뀌는 사고가 나기 쉬움.

## 5-2. 코드 보기(View Code)
- 의미: 엑셀을 표 계산 도구가 아니라 VBA 기반 자동화/프로그래밍 환경으로 전환해 보는 기능이다.
- 내용: VBA 코드, 이벤트 프로시저, 자동 실행 로직 등을 확인/작성한다.

## 5-3. 시트 보호 작업(Protect Sheet)
- 의미: “권한(가능/불가능 행동)”을 통제한다.
- 예: 셀 편집 허용 여부, 행/열 삽입·삭제 허용 여부, 개체 수정 허용 여부 등을 세부적으로 설정한다.


# 📌 6. 데이터 표(Data Table) 결과 셀의 {=TABLE(,A2)} 의미

## 6-1. 중괄호 { }의 의미
- 사용자가 일반적으로 직접 입력한 수식이 아니라, 엑셀이 데이터 표 기능을 통해 생성한 배열/특수 수식 형태임을 나타낸다.
- 데이터 표 영역의 일부 셀만 임의로 수정하기 어렵다.

## 6-2. TABLE 함수의 의미
- TABLE은 일반 워크시트 함수처럼 쓰기 위한 것이 아니라, ‘가정 분석(What-If)’의 데이터 표 계산을 수행하는 내부용 함수에 가깝다.
- 입력 값을 바꿔가며 결과 수식을 반복 계산해 표로 채우는 역할을 한다.

## 6-3. (, A2)가 뜻하는 것: 1변수 데이터 표
- 쉼표 앞 인수(행 입력 셀)가 비어 있음 → 행 입력 셀 없음
- 쉼표 뒤 인수(열 입력 셀)가 A2 → 열 입력 셀은 A2
- 의미: 데이터 표가 A2에 값들을 순서대로 대입해가며 결과를 계산하고, 각 결과를 셀에 표시한다.

## 6-4. 핵심 정리
- {=TABLE(,A2)}는 “A2를 입력 셀로 사용하는 1변수 데이터 표의 결과 셀”이라는 표시다.


# 📌 7. 부분합(Subtotal)에서 사용자 지정 수식을 못 만드는 이유

## 7-1. 부분합의 본질
- 부분합은 셀 계산 기능이 아니라, 그룹 경계에 “부분합 행을 삽입”하여 자동 집계를 만드는 ‘구조 변경’ 기능이다.

## 7-2. 사용자 지정 수식을 막는 설계 이유
- 부분합은 행 삽입/삭제, 그룹 크기 변화 등으로 데이터 구조가 자주 변한다.
- 임의 수식은 참조 범위/의미가 쉽게 흔들려 오류·혼란이 커진다.
- 그래서 엑셀은 의미가 안정적인 집계 함수(SUM, AVERAGE, COUNT 등)만 허용한다.

## 7-3. 설계 철학
- 자유로운 표현력보다 “결과의 안정성/재현성”을 우선한 기능 설계이다.


# 📌 8. VBA 코드에서 ‘기울임꼴(Italic)’을 의미하는 부분

## 8-1. 기울임꼴을 켜는 코드
- 기울임꼴은 글꼴(Font) 속성이다.
- 해당 코드는 다음 줄이 기울임꼴 적용을 의미한다.
  - Selection.Font.Italic = True

## 8-2. 자주 헷갈리는 줄(정렬과의 구분)
- .VerticalAlignment = xlCenter 는 “세로 가운데 정렬”이다.
- 정렬(Alignment)은 글꼴 모양(Italic)과 별개 층위다.

## 8-3. 핵심 구분
- 모양(기울임/굵게/밑줄 등) → Font.*
- 위치(가로/세로 정렬 등) → Alignment 관련 속성


# 📌 9. 엑셀 사용자 지정 표시 형식 [h]:mm 의미

## 9-1. h와 mm
- h: hour(시간)
- mm: minute(분)

## 9-2. 대괄호 [ ]의 핵심 기능
- 일반적인 h:mm는 24시간(1일) 단위로 ‘시계처럼’ 표시되어 24시간을 넘기면 다시 0부터 표시될 수 있다.
- [h]:mm에서 [h]는 “누적 시간을 시간 단위로 표시(24시간 제한 해제)”를 의미한다.

## 9-3. 의미 요약
- [h]:mm = “24시간을 넘어가는 누적 시간을 시:분으로 표시”
- 예시 개념:
  - 누적 27시간 30분을 h:mm로 보면 3:30처럼 보일 수 있으나,
  - [h]:mm로 보면 27:30처럼 누적 시간이 그대로 표시된다.

## 9-4. 설계 관점(철학)
- h:mm는 ‘시계 시간’ 관점,
- [h]:mm는 ‘경과/누적 시간(근무, 작업 소요시간)’ 관점이다.
