# 📌 1. Access 폼과 컨트롤의 기본 개념

## 1-1. 폼(Form)의 역할

- 폼은 **데이터를 입력·수정·검색·조작**하기 위한 화면이다.
- 사용자가 텍스트를 입력하고, 콤보 상자에서 값을 선택하고, 버튼을 눌러 작업을 실행하는 등 **상호작용(UI)** 이 일어나는 공간이다.
- 폼은 기본적으로 **실시간으로 테이블/쿼리와 연결된 데이터바운드 UI** 이다.

## 1-2. 컨트롤(Control)의 개념

- 폼·보고서 위에 올려지는 **모든 개별 UI 요소**를 컨트롤이라고 한다.
  - 예: 텍스트 상자, 콤보 상자, 체크 상자, 명령 단추, 레이블 등
- 컨트롤은 크게 두 축으로 이해한다.
  1. **데이터와 연결되는가?** (바운드 / 언바운드)
  2. **사용자와 상호작용하는가?** (입력용 / 표시용 / 버튼 등)

- Access의 폼은 “테이블의 필드를 화면의 컨트롤과 1:1로 매칭해서, 사용자가 바로 데이터를 만질 수 있게 하는 것”이 핵심 철학이다.

# 📌 2. 컨트롤 원본과 바운드/언바운드 컨트롤

## 2-1. 컨트롤 원본(Control Source)

- **컨트롤에 어떤 데이터를 연결할지, 또는 어떤 식(formula)을 사용할지 지정하는 속성**이다.
- 대표 형태
  - 특정 필드에 직접 연결  
    - 예: 컨트롤 원본 = `CustomerName`  
      → 현재 레코드의 `CustomerName` 필드 값을 표시·수정
  - 수식 사용  
    - 예: 컨트롤 원본 = `= [수량] * [단가]`
    - 예: 컨트롤 원본 = `=Date()`  
      → 계산된 값을 보여주며, 별도의 필드에 저장되지는 않는다.
- 컨트롤 원본은 **“이 컨트롤이 데이터베이스와 어떻게 연결될지 정의하는 연결선”** 이다.

## 2-2. 바운드 컨트롤(Bound Control)

- 컨트롤 원본이 테이블/쿼리의 **필드명**으로 지정된 컨트롤.
- 특징
  - 레코드를 이동하면 해당 필드 값이 자동으로 표시된다.
  - 사용자가 값을 수정하면 **실제 테이블 데이터가 변경**된다.
- 예: 텍스트 상자에 컨트롤 원본 = `고객명`
- 철학적으로:
  - 바운드 컨트롤은 테이블 속 “셀”을 화면 위로 끌어올린 것과 같다.
  - Access의 폼이 **데이터 입력 화면을 자동 생성**할 수 있는 이유가 바로 이 바인딩 구조 덕분이다.

## 2-3. 언바운드 컨트롤(Unbound Control)과 계산 컨트롤

- 컨트롤 원본이 **비어 있거나 필드에 직접 연결되지 않은 경우** 언바운드 컨트롤이다.
  - 예: 설명 문구용 레이블, 버튼, 빈 텍스트 상자, 수식만 넣은 상자 등
- 계산 컨트롤(calculated control)
  - 컨트롤 원본에 **필드가 아니라 식만 들어 있는 경우**를 말한다.
  - 예: `=Sum([Amount])`, `=Now()`
  - 화면에서 계산 결과만 표현하며, 테이블에 따로 저장하지 않는다.
- 철학:
  - **저장할 값(필드)** 과 **계산해서 보여줄 값(컨트롤)** 을 분리하는 정규화 원칙을 유지한다.
  - 데이터는 테이블에, 표현과 계산은 컨트롤에서 담당하도록 설계되어 있다.

# 📌 3. 레이블(Label) 컨트롤의 의미

## 3-1. 레이블의 본질

- 레이블은 **고정 텍스트를 보여주는 컨트롤**, 즉 “이름표(label)” 역할을 한다.
- 사용자가 값을 입력하거나 수정할 수 없으며, 데이터와도 직접 연결되지 않는다.
- 컨트롤 원본 속성이 없고, 단지 **표시할 문자열**(캡션)을 가진다.

## 3-2. 역할과 철학

- 주된 역할:
  - 텍스트 상자 옆에 `이름:`, `전화번호:`, `주문합계` 같은 **설명**을 붙여준다.
- 철학적으로:
  - 레이블은 “데이터”가 아니라 **데이터의 의미를 설명하는 언어적 표식(sign)** 이다.
  - 코드에서의 주석(comment)처럼, 데이터 구조를 사람이 이해하기 쉽게 만드는 역할을 한다.
  - Access가 텍스트 상자를 만들 때 자동으로 레이블을 같이 만드는 이유도,  
    “사용자가 이 필드가 무엇인지 잊지 않도록 의미를 부여하기 위해서”이다.

# 📌 4. 텍스트 상자와 콤보 상자, 그리고 Shift + 방향키

## 4-1. 텍스트 상자(Text Box) 컨트롤

- **데이터를 직접 입력하거나 보여주는 가장 기본 컨트롤**이다.
- 바운드 텍스트 상자
  - 컨트롤 원본에 필드명을 지정한 경우
  - 사용자가 입력하면 테이블 데이터가 바로 바뀐다.
- 언바운드 텍스트 상자
  - 컨트롤 원본이 비어 있거나 수식만 있는 경우
  - 검색창, 임시값 입력, 계산 결과 표시 등에 사용
- 설계 철학:
  - 텍스트 상자는 인간과 데이터가 직접 만나는 “입력 창구”이다.
  - 데이터 입력 → 검증 → 저장의 흐름이 가장 직접적으로 구현되는 구조이다.

## 4-2. 콤보 상자(Combo Box) 컨트롤

- 콤보 상자는 “입력이 아니라 **선택으로 데이터 입력을 유도하는 컨트롤**”이다.
- 핵심 속성
  - Row Source: 목록에 표시될 값들의 집합
  - Bound Column: 실제로 저장되는 값
  - Control Source: 이 콤보 상자에서 선택된 값이 저장될 필드
- 예: 학과 선택 콤보
  - 표시값: 간호학과, 기계공학과, 전자공학과 …
  - 저장값: $1, 2, 3$ (학과ID)
- 철학:
  - 사람에게 모든 값을 직접 타이핑하게 하면 **오타와 불일치(데이터 무결성 깨짐)** 가 발생한다.
  - 콤보 상자는 사용자가 **미리 정의된 값들만 선택**하게 해서,  
    참조 무결성(예: 외래키)과 데이터 품질을 지키도록 설계된 도구이다.

## 4-3. Shift + 방향키의 개념적 의미

- Shift 키는 “범위를 확장한다”는 의미를 가진 보조키(modifier key)이다.
- 방향키는 “이동”을 의미한다.
- 따라서 **Shift + 방향키 = 이동하면서 선택 범위를 확장** 이라는 의미가 된다.
  - 텍스트 편집기: 글자를 하나씩/한 줄씩 선택
  - 파일 탐색기: 연속된 파일들을 선택
  - 엑셀/Access 디자인: 연속된 셀·컨트롤 범위를 선택
- 철학:
  - 사람의 사고가 **연속적인 리스트·라인을 하나의 덩어리로 인식**한다는 점을 반영한 키 설계이다.
  - Ctrl이 “비연속적 선택(여기저기 따로 선택)”에 가깝다면,  
    Shift는 “연속된 구간을 한꺼번에 선택”하는 개념이다.

# 📌 5. Access 객관식·단답식 문제에 나온 속성들의 근본 원리

## 5-1. 문제 1 — 캡션(Caption)과 단축키(Alt + T)

> 명령 단추 컨트롤의 (  ) 속성에 `&T`를 입력하면 Alt + T를 눌렀을 때 해당 폼의 해당 명령 단추 컨트롤로 포커스를 이동시킬 수 있다.  
> 정답: 캡션(Caption)

- UI 전통에서 `&` 기호는 **단축키(Access Key)** 를 뜻한다.
  - `&T` → 화면에 `T`가 밑줄이 그어진 상태로 보이고, Alt + T로 포커스를 이동
- 단축키는 “보여지는 글자”에 연결해야 의미가 있으므로,  
  **사용자에게 표시되는 이름 속성인 캡션(Caption)에 `&T`를 넣는다.**
- 철학:
  - 키보드로도 UI를 빠르게 조작할 수 있도록 해, 마우스 없이도 작업 흐름을 유지하게 하는 설계이다.
  - 특히 폼에서 명령 단추를 많이 사용할 때 **손이 키보드에서 떨어지지 않도록** 돕는다.

## 5-2. 문제 2 — 컨트롤 원본(Control Source)

> (  ) 속성은 컨트롤에 연결할 데이터를 지정한다.  
> 정답: 컨트롤 원본

- 이미 정리했듯, 컨트롤 원본은 컨트롤과 필드(또는 식)를 연결하는 핵심 속성이다.
- 문제에서 “연결할 데이터를 지정한다”라는 문장은 **바운드 컨트롤을 정의하는 말**이라고 보면 된다.

## 5-3. 문제 3 — 탭 순서(Tab Order)

> (  )는 폼에서 데이터를 입력받는 컨트롤의 순서를 정할 때 사용하는 속성이다.  
> 정답: 탭 순서

- Tab 키는 폼의 입력 흐름을 제어하는 키이다.
- 탭 순서는 “사용자가 Tab을 눌러 입력할 때, **포커스가 이동하는 논리 순서**”를 의미한다.
- 철학:
  - 데이터 입력 업무는 대량·반복이 많기 때문에,  
    **마우스를 쓰지 않고도 자연스러운 순서로 입력**할 수 있도록 설계되었다.
  - 탭 순서가 엉망이면 사용자는 “폼이 나랑 싸우는 느낌”을 받게 된다.

## 5-4. 문제 6 — 컨트롤 원본과 누적합계(Running Sum) 속성

> 거래처별로 그룹이 설정된 ‘매출 내역’ 보고서에서, 본문 영역의 `txt순번` 텍스트 상자 컨트롤에 거래처별 매출 순번(1,2,3,…)을 표시하려고 한다.  
> 올바른 속성 설정:  
> **컨트롤 원본 = 1, 누적합계 = 그룹**

- 보고서는 레코드를 한 줄씩 출력하면서 텍스트 상자를 계산한다.
- `컨트롤 원본 = 1`로 설정하면 **각 레코드마다 기본값 1**이 들어간다.
- 여기에 누적합계(Running Sum) 속성을 사용하면:
  - 첫 레코드: $1$
  - 둘째 레코드: $1 + 1 = 2$
  - 셋째 레코드: $2 + 1 = 3$
- 누적합계를 “그룹”으로 설정하면:
  - 거래처 A 그룹: $1, 2, 3, \dots$
  - 거래처 B 그룹으로 넘어가면 다시 $1$부터 시작
- 철학:
  - 순번은 **저장하는 값이 아니라, 보고서를 출력할 때 계산해서 보여줄 값**이라는 관점이다.
  - 데이터는 테이블에, 순번·합계·누적값 같은 “표현용 정보”는 보고서 컨트롤에서 계산하는 구조를 선호한다.

# 📌 6. 하위폼(Subform)의 개념과 철학

## 6-1. 겉모습: 폼 안에 있는 또 다른 폼

- UI 구조만 보면 “메인 폼 안에 들어간 또 하나의 폼”이다.
- 하지만 단순히 폼을 박아넣은 것이 아니라, **관계형 데이터(1:N)를 표현하기 위한 특수한 폼**이다.

## 6-2. 본질: 부모-자식 관계(1:N)를 표현하는 장치

- 예시:
  - 고객(부모) : 주문(자식)
  - 주문(부모) : 주문 상세(자식)
- 메인 폼: 부모 테이블 레코드 표시
- 하위폼: 현재 선택된 부모 레코드에 속한 자식 레코드 리스트 표시
- 이를 위해 다음 두 속성을 사용한다.
  - Link Master Fields: 부모 쪽 키 필드
  - Link Child Fields: 자식 쪽 외래키 필드
- 이 설정을 통해:
  - 메인 폼에서 고객 ID가 $10$인 레코드를 보고 있다면  
    하위폼에는 고객 ID가 $10$인 주문들만 자동으로 보인다.

## 6-3. 설계 철학

- **관계형 데이터 모델을 UI 구조로 그대로 드러낸다.**
  - $1:N$ 관계 → 메인 폼과 하위폼의 구성으로 시각화
- 개발자가 복잡한 조인·필터 코드를 쓰지 않아도,  
  Access가 자동으로 자식 레코드를 필터링해서 보여 준다.
- 사용자는 메인 폼에서 부모 정보를 보고, 하위폼에서 여러 자식 행을 추가/수정/삭제할 수 있으므로,
  **관계형 데이터 입력을 자연스럽게 수행**하게 된다.

# 📌 7. 폼(Form)과 보고서(Report)의 차이

## 7-1. 존재 목적의 차이

- 폼(Form)
  - 목적: **입력·수정·검색·조작**을 위한 작업 화면
  - 데이터와 상호작용하는 UI
  - 버튼, 콤보 박스, 텍스트 입력, 이벤트 처리 등 “행동 중심”
- 보고서(Report)
  - 목적: **출력·요약·정리·배치**를 위한 결과 화면
  - 인쇄/배포용 문서
  - 그룹별 합계, 누적합계, 페이지 번호 등 “표현 중심”

## 7-2. 상호작용성

| 구분 | 폼(Form) | 보고서(Report) |
|------|---------|----------------|
| 데이터 입력/수정 | 가능 | 불가능 |
| 버튼 클릭 이벤트 | 활발 | 제한적 |
| 콤보 상자 선택 | 가능 | 거의 없음 |
| 테이블 업데이트 | 직접 반영 | 읽기 전용 |
| 인쇄 최적화 | 약함 | 강함 |

- 폼은 **살아 있는 작업 공간**,  
- 보고서는 **고정된 출력물**이라고 이해하면 된다.

## 7-3. 레이아웃과 그룹 기능의 차이

- 보고서
  - 그룹 헤더/푸터, 페이지 헤더/푸터, 정렬 및 그룹 지정, 누적합계 등  
    문서 형태로 데이터를 표현하기 위한 기능이 매우 강력하다.
- 폼
  - 데이터 탐색과 입력에는 강하지만,  
    문서 인쇄용 레이아웃(페이지 관리 등)에는 적합하지 않다.

## 7-4. 설계 철학 요약

- 폼의 철학
  - “사람이 데이터를 잘 만지게 하자.”
  - 탭 순서, 버튼, 콤보 상자, 검증 규칙 등 **업무 흐름 중심 설계**
- 보고서의 철학
