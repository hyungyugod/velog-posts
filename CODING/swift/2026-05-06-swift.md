## 📌 1. Phase 1-3 ~ 1-5 · 입력·경계·카메라 (이동 시스템 완성)

### 1-1. Phase 1-3 · PlayerNode + 반투명 D-Pad + dt 이동

> 💡 자동 왕복 박스 → *사용자 입력으로 움직이는* 캐릭터로 교체. 게임이 처음으로 게임답게 반응.

#### 1) 두 좌표계가 동시 등장
- `worldNode`(월드 절대 좌표) 자식 = PlayerNode → 맵 안 절대 위치
- `cameraNode`(카메라 상대 좌표) 자식 = DPadNode → 화면 우하단 *고정*
- 같은 (320, 200)이라도 부모에 따라 의미가 달라짐 → JPA Entity vs DTO 분리와 같음

#### 2) Swift 핵심 패턴
- `final class` + `private(set) var` + `required init?(coder:)` + `fatalError` 보일러플레이트
- DPadNode는 `SKNode` 상속 + `isUserInteractionEnabled = true`(기본 false라 명시 필수)
- DPadNode → PlayerNode *직접 참조 금지* — GameScene 경유 (단방향 의존)

```swift
final class PlayerNode: SKSpriteNode {
    var currentDirection: CGVector = .zero
    func update(deltaTime: TimeInterval) {
        position.x += currentDirection.dx * GameConfig.playerBaseSpeed * CGFloat(deltaTime)
        position.y += currentDirection.dy * GameConfig.playerBaseSpeed * CGFloat(deltaTime)
    }
}
```

#### 3) 핫픽스 — `scaleMode = .resizeFill`
- iPhone 17 viewport(852×393)와 scene size(1024×768) 비율 차로 D-Pad가 화면 밑 잘림
- `.aspectFill` → `.resizeFill` 한 줄 변경으로 모든 디바이스 대응
- *고정 해상도 게임*이 아니면 cameraNode 자식 UI는 거의 항상 `.resizeFill`이 정답

### 1-2. Phase 1-4 · 월드 경계 + 카메라 클램핑

> 💡 박스가 검은 영역으로 못 나가게 + 카메라가 맵 끝에서 멈춤. 처음으로 *공간*이 정의됨.

#### 1) `max(min(...))` 클램프 관용구
- Swift 표준에 `clamp` 없음 → `max(lower, min(upper, value))`로 합성
- PlayerNode 안 자체 클램프 = *도메인 자체 validation* (Spring `@Max` 비유)
- 책임 분리: PlayerNode는 *맵 정보*만, GameScene은 *viewport 정보*만

```swift
position.x = max(halfW, min(GameConfig.mapWidth  - halfW, position.x))
position.y = max(halfH, min(GameConfig.mapHeight - halfH, position.y))
```

#### 2) viewport vs map 분기는 *영구 코드*
- iPhone 17 가로: viewport(852) > map(640) → 맵 중앙 고정
- iPhone 17 가로: viewport(393) < map(400) → 일반 클램프
- 디바이스마다 동작이 자연스럽게 달라지는 영구 분기

#### 3) 클램프 순서 — 이동 → 검증
- 검증 → 이동 순이면 한 프레임 늦어짐. 명령형 순서가 의미 결정

### 1-3. Phase 1-5 · 카메라 정책 재설계 + 맵 확장

> 💡 1-4 클램핑을 *덜어내고* 박스를 항상 화면 정중앙에 둠 (드론 follow). 맵도 48×24=960×480으로 확장.

#### 1) 디자인 정책의 *번복은 정상*
- 1-4 클램핑은 횡스크롤 게임에 맞음 (마리오, 메트로이드)
- GanhoMusic은 *탑다운 회피 게임* → 사방 시야 균등이 본질 → 드론 follow가 맞음
- 회고에 *왜 뒤집었는지* 남기면 미래의 자신/Claude가 같은 실수 안 반복

#### 2) 코드 *덜어내기*가 더 어렵다
- 24줄 헬퍼 함수 + 분기 → `cameraNode.position = player.position` 한 줄로 회귀
- *MARK 섹션 통째로* 한 번에 삭제 (부분 삭제 시 dangling brace)

#### 3) `static let` 파생값의 자동 갱신
- `mapColumns: 32 → 48` 한 곳 변경으로 `mapWidth` 파생식이 `640 → 960` 자동 갱신
- 호출부(자체 클램프, corner 마커, setupPlayer) 코드 변경 0건

```swift
static let mapColumns: Int = 48          // 변경
static let mapWidth: CGFloat = tileSize * CGFloat(mapColumns)  // 자동 960
```

## 📌 2. Phase 2-1 ~ 2-2 · 물리와 공간 (Physics 활성화)

### 2-1. Phase 2-1 · 외곽 벽 시각화

> 💡 검은 우주였던 맵 가장자리에 벽을 그어서 *공간감 완성*. corner 마커는 자연 폐기.

#### 1) `SKSpriteNode`로 얇고 긴 벽 4개
- 두께 1tile(20pt), 맵 *바깥쪽*에 배치 (벽 안쪽이 mapWidth/Height에 정확히 닿음)
- 함수 4번 풀어쓰기 ❌ → `WallSpec` 지역 struct + `for` 루프로 1회 작성

```swift
struct WallSpec { let size: CGSize; let position: CGPoint }
let walls: [WallSpec] = [/* top, bottom, left, right */]
for spec in walls {
    let wall = SKSpriteNode(color: .ganhoPaper, size: spec.size)
    wall.position = spec.position
    worldNode.addChild(wall)
}
```

#### 2) `.ganhoPaper` 토큰 재활용 + alpha로 시각 계층 분리
- D-Pad: alpha 0.3 (반투명 UI) / 외곽 벽: alpha 1.0 (솔리드 게임 오브젝트)
- 같은 색이지만 *역할*이 자동 분리. 새 ColorTokens 추가 0건 (1-1 자산 보존)

#### 3) 임시물의 *폐기 시점* 명시
- corner 마커는 1-2부터 학습 노트에 "Phase 2 맵 타일 도입 시 폐기" 명시
- Spring `// TODO`보다 *시점*까지 적는 게 강력 — SPEC이 그 역할

### 2-2. Phase 2-2 · 중앙 기둥 + SKPhysicsBody 첫 도입

> 💡 1-1에서 만들어둔 PhysicsCategory가 *드디어* 활성화. 박스가 유령에서 *몸을 가진* 캐릭터로 진화.

#### 1) PhysicsBody 핵심 속성 9가지
- `isDynamic`: true(player) / false(벽·기둥) — 누가 움직이나
- `categoryBitMask`(나의 정체) / `collisionBitMask`(막힘) / `contactTestBitMask`(알림)
- `allowsRotation = false` (박스 데굴데굴 방지)
- `friction/restitution/linearDamping = 0` (D-Pad 떼면 즉시 정지)
- `physicsWorld.gravity = .zero` (탑다운 — 씬 한 줄로 끝)

```swift
let body = SKPhysicsBody(rectangleOf: size)
body.isDynamic = true
body.allowsRotation = false
body.friction = 0; body.restitution = 0; body.linearDamping = 0
body.categoryBitMask    = PhysicsCategory.player
body.collisionBitMask   = PhysicsCategory.wall
body.contactTestBitMask = 0   // Phase 2-3에서 .note 추가
physicsBody = body
```

#### 2) velocity 기반 이동 = *명령형 → 선언형* 전환
- 1-3: `position += dt * speed * dir` (위치 직접 통제)
- 2-2: `physicsBody?.velocity = speed * dir` (의도 표명, 엔진이 위치 통제)
- 충돌 시 엔진이 자동으로 velocity 0 처리 → 1-4 자체 클램프 4줄 *제거* (단일 책임)

#### 3) PhysicsCategory = *권한 비트마스크*
- 1-1에서 정의만 해두고 5단계 sleep 자산이 *처음 활성화*
- Spring Security `GrantedAuthority`의 비트 버전

## 📌 3. Phase 2-3 · 음표 스폰 + 충돌 *알림*

> 💡 collision(2-2)과 contact(2-3) 분리 — 같은 SKPhysicsBody, 다른 비트마스크. 게임이 *공간*에서 *게임*으로 진화.

### 3-1. NoteNode + 자동 스폰

#### 1) `collision = 0` + `contactTest = .player` (통과+알림)
- 음표는 박스를 *막지 않고* 통과 → `collisionBitMask = 0`
- 그러나 *접촉 알림*은 받음 → `contactTestBitMask = .player`
- `isDynamic = false` (음표가 밀려나면 안 됨, 수집 위치 예측 가능성)

```swift
body.isDynamic = false
body.categoryBitMask    = PhysicsCategory.note
body.collisionBitMask   = 0                       // 통과
body.contactTestBitMask = PhysicsCategory.player  // 알림
```

#### 2) `SKAction.repeatForever(sequence([wait, run]))` 패턴
- `Timer` 금지 룰 — Foundation 시간 vs 씬 시간이 다름 (일시정지 시 Timer는 별개 흐름)
- `[weak self]` 캡처 + `withKey: "spawnNotes"` 명시 → 정확히 끄기 위한 보험
- Spring `@Scheduled(fixedRate=1500)`과 동치, 호스트만 다름

#### 3) `enumerateChildNodes(withName:)` + `CGFloat.random(in:)`
- 동시 5개 제한: `currentNoteCount() >= noteMaxConcurrent`이면 skip
- 외곽 1tile 마진 + 중앙 기둥 manhattan 60pt 회피

### 3-2. SKPhysicsContactDelegate

#### 1) `didBegin(_:)` 비대칭 식별 패턴
- 두 body 중 어느 쪽이 note인지 비트마스크로 식별 후 노드 제거
- *대칭*이라 한쪽만 contactTest 설정해도 콜백 1회

```swift
func didBegin(_ contact: SKPhysicsContact) {
    let noteBody: SKPhysicsBody?
    if contact.bodyA.categoryBitMask == PhysicsCategory.note { noteBody = contact.bodyA }
    else if contact.bodyB.categoryBitMask == PhysicsCategory.note { noteBody = contact.bodyB }
    else { noteBody = nil }
    guard let noteNode = noteBody?.node else { return }
    score += 1
    noteNode.run(.removeFromParent())   // 즉시 X — 다음 프레임 안전 제거
}
```

#### 2) `note.run(.removeFromParent())` — *다음 프레임* 정리
- `removeFromParent()` 즉시 호출은 같은 프레임 다른 콜백에서 *해체된 노드 참조* 위험
- SKAction으로 위임 → 프레임 종료 후 정리 = 안전 (Spring `@Async` 비유)

### 3-3. 비교표 — collision vs contact

| 비트마스크 | 의미 | Spring 비유 | 2-2 | 2-3 |
|---|---|---|---|---|
| `category` | 나의 정체 | DI 빈 type | wall/player | + note |
| `collision` | 물리적 막힘 | DB CHECK 제약 | wall ↔ player | 변경 없음 |
| `contactTest` | 충돌 알림 | ApplicationEvent | 모두 0 | player ↔ note 활성 |

## 📌 4. Phase 2-4 ~ 2-5 · HUD + 콤보 (보상 시스템)

### 4-1. Phase 2-4 · 점수 라벨 + 45초 타이머 + 게임오버

> 💡 점수가 *눈에 보임* + 시간 만료로 게임이 끝남. 게임이 *공간*에서 *세션*으로 진화.

#### 1) HUDNode = `SKNode` 컨테이너 패턴
- `SKNode`는 *시각 자체는 없음* (JPanel 같은 레이아웃 컨테이너)
- 라벨 2개 묶음 + `update(score:remainingTime:)`로 외부 갱신만 받음
- `cameraNode` 자식 → 화면 좌상단 고정 (D-Pad 우하단과 *대각 대칭*)

```swift
final class HUDNode: SKNode {
    private let scoreLabel: SKLabelNode
    private let timeLabel: SKLabelNode
    func update(score: Int, remainingTime: TimeInterval) {
        scoreLabel.text = "🎵 \(score)"
        let seconds = max(0, Int(ceil(remainingTime)))
        timeLabel.text = String(format: "⏱ 00:%02d", seconds)
    }
}
```

#### 2) `SKLabelNode` 정렬 모드 *명시* 필수
- 기본값 `center`/`baseline` → 점수 0→9999 변할 때 글자 폭만큼 위치 흔들림
- `horizontalAlignmentMode = .left` + `verticalAlignmentMode = .top` 명시

#### 3) `dt` 카운트다운 (Timer 금지)
- `remainingTime = max(0, remainingTime - dt)` + `if remainingTime <= 0 { endGame() }`
- *카운트다운 직후 즉시 검사 + early return* — update 끝에서 검사하면 마지막 음표가 게임오버 후 생성

#### 4) `endGame()` — 상태 전환 + 사이드 이펙트 일괄
- `gameState = .gameOver` + `removeAction(forKey: "spawnNotes")` + player 정지
- player 정지는 *둘 다*: `currentDirection = .zero` + `physicsBody?.velocity = .zero`
  - 의도(다음 프레임)와 상태(현 프레임 잔여 운동)의 분리

#### 5) `ceil` 사용 = *체감 자연스러움 우선*
- 44.7초 → 라벨 "44"면 사용자가 "1초 사라짐" 느낌
- `ceil`로 올림 → 시작 직후 "45" 1초간 보임. 수학적 정확보다 UX 우선

### 4-2. Phase 2-5 · 콤보 시스템 + 점수 ×2

> 💡 2.5초 안 연속 수집 시 콤보 누적, 3콤보부터 점수 ×2. 게임이 *반복 수집*에서 *연속 압박이 보상*인 구조로 진화.

#### 1) Optional 회피 = *센티널 값 + 가드*
- `lastCollectAt: TimeInterval?` 대신 `= 0` + `combo > 0` 가드
- 첫 수집 전엔 만료 검사 자체 진입 안 함 → 0과 비교 없음
- 언어가 권장한다고 늘 옳은 건 아님 — *논리적 가드*가 더 깔끔할 때 있음

```swift
private var combo: Int = 0
private var lastCollectAt: TimeInterval = 0   // 0 = "아직 수집 0건"

// update(_:) 만료 검사
if combo > 0, currentTime - lastCollectAt > GameConfig.comboWindow {
    combo = 0
}
```

#### 2) 점수 분기 = *임계 + 단가 모두 상수*
- `combo >= 3 ? 2 : 1` ❌ → 매직 넘버
- `combo >= comboBonusThreshold ? scorePerNoteCombo : scorePerNote` ⭐ 룰이 *데이터*

```swift
score += combo >= GameConfig.comboBonusThreshold
    ? GameConfig.scorePerNoteCombo   // 2
    : GameConfig.scorePerNote         // 1
```

#### 3) `lastUpdateTime` 재활용 = *기존 인프라 부수 사용*
- 2-4의 dt 계산용 변수를 콤보 만료 검사·didBegin 시점 비교에 그대로 사용
- 새 변수 0개. Spring DI 빈을 여러 컴포넌트가 공유하는 패턴

#### 4) `alpha` 분기 vs `isHidden` 토글
- `alpha`는 GPU 셰이더 단계 → 비용 0
- `isHidden`은 SpriteKit 노드 트리 갱신 → 60Hz 토글에 비용 발생
- 콤보 < 2면 `comboLabel.alpha = 0` (조건부 표시)

#### 5) 시그니처 *확장* — 명명 인자가 무료 회귀 검사
- `update(score:remainingTime:)` → `update(score:remainingTime:combo:)`
- 컴파일러가 호출처 빠진 인자 자동 식별
- Java 익명 인자라면 못 잡음 → Swift 명명 인자 강제의 보너스

## 📌 5. 종합 비교표 + 핵심 패턴

### 5-1. 비교표 — 8단계 변경량 vs 의미량

| Phase | 변경량 | 의미량 | 핵심 |
|---|---|---|---|
| 1-3 | 신설 2 + 수정 2 | 사용자 입력 시작 | dt 보간 + 두 좌표계 |
| 1-4 | 수정 2 (4줄 + 헬퍼) | 공간 정의 | clamp 관용구 |
| 1-5 | 삭제 24줄 + 값 2개 | 카메라 정책 번복 | 파생값 자동 갱신 |
| 2-1 | 함수 1폐 + 1신 | 시각 공간감 완성 | for 루프 + 색 alpha |
| 2-2 | 수정 2 | 물리 활성화 | velocity + PhysicsBody |
| 2-3 | 신설 1 + 수정 3 | 게임이 됨 | collision ≠ contact |
| 2-4 | 신설 1 + 수정 2 | 세션 시작·끝 | dt 카운트다운 + endGame |
| 2-5 | 수정 3 | 연속 보상 | rolling window + 임계 분기 |

### 5-2. 반복 등장한 핵심 패턴

#### 1) 명령형 → 선언형 전환
- 1-3 dt 직접 이동 → 2-2 velocity 표명
- 2-3 SKAction 스폰 루프 → 2-4 removeAction 키 정책

#### 2) 도메인 vs 씬 책임 분리
- PlayerNode (맵 정보) ↔ GameScene (viewport 정보)
- Service (값 보유) ↔ Controller (값 노출, HUDNode)

#### 3) 상수화 + 시점 명시
- 모든 매직 넘버 → GameConfig
- 임시물에 *폐기 시점*까지 명시 (Spring `// TODO`보다 강력)

#### 4) 1-1 자산의 *지연 활성화*
- PhysicsCategory: 1-1 정의 → 2-2 wall 활성 → 2-3 note 활성
- ColorTokens (`.ganhoPinkNote`): 1-1 정의 → 2-3 활성
- GameState (`.gameOver`): 1-1 정의 → 2-4 활성
- *인프라 먼저, 활성화 나중* = Spring `@ConfigurationProperties` 패턴
