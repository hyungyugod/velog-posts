# 📌 1. 함수적 종속과 “종속자가 모든 결정자에 종속된다”의 의미

## 1-1. 함수적 종속과 결정자·종속자

- **함수적 종속(Function Dependency)**  
  - 어떤 릴레이션에서 속성 집합 $X$의 값이 정해지면 속성 집합 $Y$의 값이 유일하게 정해질 때  
    $X \rightarrow Y$ 라고 함.
- **결정자(Determinant)**  
  - 다른 속성 값을 **결정하는 속성(집합)**.  
  - 예: `학번 → 이름, 학과` 에서 `학번`이 결정자.
- **종속자(Dependent)**  
  - 결정자에 의해 **값이 결정되는 속성**.  
  - 위 예에서는 `이름`, `학과`가 종속자.

## 1-2. “모든 결정자에 종속된다” = 완전 함수적 종속

- 복합키 $(A, B)$에 대해
  - $(A, B) \rightarrow C$ 이고  
  - $A \rightarrow C$, $B \rightarrow C$ 가 **성립하지 않을 때**  
  → $C$는 $(A, B)$에 대해 **완전 함수적 종속(Full Functional Dependency)**.
- 의미:
  - 종속자가 키의 **일부가 아니라 전체**에 의해서만 결정된다.
  - 즉 “종속자가 모든 결정자에 종속된다” =  
    복합키의 **모든 구성 요소가 함께 있어야** 그 종속자를 결정할 수 있다는 뜻.
- 중요성(2NF와의 관계):
  - **제2정규형(2NF)**는 기본키의 **부분집합에만 종속(부분 함수적 종속)**되는 속성을 제거하는 정규형.
  - 따라서 “모든 결정자(복합키 전체)에 종속”될수록 2NF 조건에 가까워짐.

# 📌 2. OSI 7계층에서 전송계층의 위치와 역할

## 2-1. OSI 7계층 구조

1. 물리(Physical)  
2. 데이터링크(Data Link)  
3. 네트워크(Network)  
4. **전송(Transport)**  
5. 세션(Session)  
6. 표현(Presentation)  
7. 응용(Application)

## 2-2. 전송계층은 “중간 계층”

- 물리·데이터링크·네트워크(1~3계층)는 **하드웨어/경로 중심 하위 계층**.
- 세션·표현·응용(5~7계층)은 **사용자/응용 중심 상위 계층**.
- 전송계층(4계층)은:
  - 하위(1~3)와 상위(5~7)를 연결하는 **중간 계층**.
  - 상위 입장에서는 “하위 계층”처럼 보이지만, OSI 전체에서 통상 말하는 “하위 계층(1~3)”에는 포함되지 않음.

## 2-3. 전송계층의 핵심 기능

- **종단 간(end-to-end) 통신 제어**
  - 프로세스 대 프로세스 통신.
  - 오류 제어, 흐름 제어, 분할/재조립.
- 대표 프로토콜:
  - **TCP**: 연결 지향, 신뢰성 보장.
  - **UDP**: 비연결, 빠르지만 신뢰성 보장 없음.

# 📌 3. POP3와 교착상태(Deadlock) 필요조건

## 3-1. POP3 (Post Office Protocol v3)

- **용도**: 메일 서버에서 클라이언트로 메일을 **다운로드**하는 응용 계층 프로토콜.
- 기본 포트: TCP 110번 (보안 POP3S: 995).
- 동작 단계:
  1. **인증(Authorization)**: `USER`, `PASS`.
  2. **처리(Transaction)**: `STAT`, `LIST`, `RETR`, `DELE` 등.
  3. **갱신(Update)**: `QUIT` 시 삭제 예약 메일 실제 삭제.
- 특징:
  - 메일을 **로컬로 내려받아/offline** 사용.
  - 서버 내 메일을 삭제하는 방식이 일반적 → 다중 기기 동기화엔 불리 (IMAP과 대비).

## 3-2. 교착상태의 4가지 필요충분 조건

- 교착상태(Deadlock): 여러 프로세스가 서로가 가진 자원을 기다리며 **영원히 대기**하는 상태.

### (1) 상호배제(Mutual Exclusion)
- 자원은 동시에 **한 프로세스만** 사용할 수 있음.
- 예: 프린터 1대를 두 프로세스가 동시에 쓸 수 없음.

### (2) 점유와 대기(Hold and Wait)
- 프로세스가 **이미 자원을 점유한 상태에서**, 추가 자원을 요청하며 **대기**.

### (3) 비선점(No Preemption)
- 할당된 자원을 OS가 **강제로 빼앗을 수 없음**.
- 자원을 가진 프로세스가 스스로 반납해야 함.

### (4) 환형 대기(Circular Wait)
- 프로세스들이 자원을 **원형으로 기다리는** 관계:
  - $P_1$은 $P_2$가 가진 자원 기다림  
  - $P_2$는 $P_3$의 자원 기다림  
  - …  
  - $P_n$은 다시 $P_1$의 자원 기다림

> 위 4가지 조건이 **모두 동시에** 만족될 때 교착상태가 발생한다  
> → 이들 조건은 교착상태의 **필요충분 조건**.

# 📌 4. 시스템 인증의 종류와 인덱스 생성 SQL

## 4-1. 시스템 인증의 3가지 분류

### 4-1-1. 소유 기반 인증 (Something you have)

- **사용자가 가진 것**으로 인증.
- 예: OTP, 보안카드, 스마트카드, 휴대폰 인증서 등.

### 4-1-2. 지식 기반 인증 (Something you know)

- **사용자가 알고 있는 정보**로 인증.
- 예: 비밀번호, PIN, 보안질문.

### 4-1-3. 생체 기반 인증 (Something you are)

- **신체적 특징**을 이용.
- 예: 지문, 홍채, 얼굴 인식, 음성.

### 4-1-4. 다요소 인증 (MFA)

- 둘 이상을 결합 (예: 비밀번호 + OTP).

## 4-2. INDEX 생성 SQL 문법

```sql
CREATE INDEX index_name
ON table_name (column1, column2, ...);
```

- **UNIQUE INDEX**: 중복 불가.
- **복합 인덱스**: 여러 컬럼에 인덱스.
- **DROP INDEX index_name;** → 인덱스 삭제.

# 📌 5. DDL·DML·DCL과 하향식 통합(깊이우선·너비우선)

## 5-1. SQL 명령어 분류

| 분류 | 의미 | 주요 명령 |
|------|------|-----------|
| DDL | 데이터 구조 정의 | CREATE, ALTER, DROP |
| DML | 데이터 조작 | SELECT, INSERT, UPDATE, DELETE |
| DCL | 권한 제어 | GRANT, REVOKE |

- DDL: 자동 커밋, 롤백 불가.
- DML: 트랜잭션 제어 가능.
- DCL: 권한 관리.

## 5-2. 하향식 통합(Top-Down Integration)

- 상위 모듈부터 하위로 테스트.

### (1) 깊이우선(Depth-First)
- 한 경로를 끝까지 통합.
- 기능 단위 검증이 빠름.

### (2) 너비우선(Breadth-First)
- 같은 레벨 모듈 먼저 통합.
- 전체 흐름 조기 검증.

# 📌 6. 디스크 스케줄링, Working Set, Mesh Network, 메모리 할당 전략

## 6-1. 디스크 스케줄링 알고리즘

| 알고리즘 | 설명 | 특징 |
|-----------|------|------|
| FCFS | 도착 순서대로 | 공평하지만 느림 |
| SSTF | 가장 가까운 요청 우선 | 평균 탐색시간 ↓, 기아 발생 가능 |
| SCAN | 왕복형(엘리베이터) | 응답시간 균형 |
| C-SCAN | 한 방향만 이동 | 응답시간 일정 |
| LOOK/C-LOOK | 요청 있는 곳까지만 이동 | 효율적 |

## 6-2. Working Set

- 일정 시간 동안 실제 참조된 페이지 집합.
- $W(t, \Delta)$ 로 표현.
- 스래싱 방지, 메모리 관리 효율 향상.

## 6-3. Mesh Network

- 노드가 상호 연결된 구조.
- **Full Mesh**: 모두 연결 / **Partial Mesh**: 일부 연결.
- 장점: 신뢰성↑ / 단점: 비용, 복잡도↑.

## 6-4. 메모리 관리 전략

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| First Fit | 첫 번째로 맞는 공간 | 빠름 | 단편화 발생 |
| Best Fit | 가장 작은 여유공간 | 낭비↓ | 작은 홀 증가 |
| Worst Fit | 가장 큰 공간 | 큰 작업 대비 | 낭비↑ |

# 📌 7. 교착상태 해결방안과 IPS

## 7-1. 교착상태 해결방안

| 방법 | 설명 | 장단점 |
|------|------|--------|
| 예방(Prevention) | 필요조건 중 하나 제거 | 자원낭비 가능 |
| 회피(Avoidance) | 안전상태 유지 (Banker’s) | 미리 요구량 필요 |
| 탐지(Detection) | 발생 후 탐지 | 오버헤드↑ |
| 회복(Recovery) | 탐지 후 해제 | 작업 손실 가능 |

## 7-2. IPS (Intrusion Prevention System)

- **침입 방지 시스템**  
  IDS는 탐지, IPS는 탐지+차단.
- 인라인 구조로 트래픽 차단 가능.
- 공격 시 패킷 폐기, 세션 종료, IP 차단 수행.

# 📌 8. 절차형 SQL(프로시저·사용자 정의 함수·트리거)과 MQTT·NFC·오버레이

## 8-1. 절차형 SQL의 3가지

### 8-1-1. 프로시저(Stored Procedure)
- 여러 SQL 문을 묶은 프로그램 블록.
- 입력/출력 매개변수 사용 가능, 반환값은 선택적.

```sql
CREATE PROCEDURE calc_bonus(IN emp_id INT)
BEGIN
    UPDATE employee
    SET bonus = salary * 0.1
    WHERE id = emp_id;
END;
```

### 8-1-2. 사용자 정의 함수(User Defined Function, UDF)
- 반드시 하나의 값을 반환.
- SELECT 문이나 WHERE 절에서 사용 가능.

```sql
CREATE FUNCTION get_tax(price DECIMAL)
RETURNS DECIMAL
BEGIN
    RETURN price * 0.1;
END;
```

### 8-1-3. 트리거(Trigger)
- INSERT, UPDATE, DELETE 등 이벤트 발생 시 자동 실행.

```sql
CREATE TRIGGER trg_update_stock
AFTER INSERT ON orders
FOR EACH ROW
BEGIN
    UPDATE product
    SET stock = stock - NEW.quantity
    WHERE id = NEW.product_id;
END;
```

## 8-2. MQTT (Message Queuing Telemetry Transport)

- IoT 기기 간 통신용 **경량 메시징 프로토콜**.
- **Publish/Subscribe** 구조, 중앙 브로커가 중계.
- **저전력, 저대역폭 환경에 적합**.

## 8-3. NFC (Near Field Communication)

- 10cm 내 근거리 무선통신.
- **보안성 높고, 속도 빠름**.
- 교통카드, 결제, 출입통제 등에 사용.

---

## 8-4. 오버레이 기법 (Overlay)

- 메모리가 부족하던 시절 프로그램 일부만 메모리에 적재하는 기법.
- **필요한 부분만 교체 적재**, 현재는 가상 메모리가 이를 대체.

# 📌 9. C언어 2차원 배열과 포인터

```c
int ar[4][2] = {1,2,3,4,5,6,7,8};
int *p = ar;
```

| 인덱스 | 값 | 표현식 |
|--------|----|--------|
| ar[0][0] | 1 | *(p + 0) |
| ar[0][1] | 2 | *(p + 1) |
| ar[1][0] | 3 | *(p + 2) |
| ar[1][1] | 4 | *(p + 3) |
| ar[2][0] | 5 | *(p + 4) |
| ar[2][1] | 6 | *(p + 5) |
| ar[3][0] | 7 | *(p + 6) |
| ar[3][1] | 8 | *(p + 7) |

- `*(p + 7)` → 8 출력.  
- `*(p + 3) + 1` → 4 + 1 = 5.  
  즉, **포인터 이동이 아니라 값에 1을 더하는 것**.

# 📌 11. PSK, BaaS, 디지털 서명, 개념 데이터 모델링, 스와핑

## 11-1. PSK (Pre-Shared Key)
- 사전 공유 키 기반 암호화.  
- Wi-Fi, VPN 등에서 사용.  
- 키 유출 시 전체 통신 노출.

## 11-2. BaaS (Blockchain as a Service)
- 클라우드 기반 블록체인 인프라 서비스.  
- 노드·합의 관리 대행, 빠른 구축 가능.  
- 의존성 증가 단점.

## 11-3. 디지털 서명 (Digital Signature)
- 무결성, 인증, 부인방지 제공.  
- 개인키로 서명 → 공개키로 검증.

## 11-4. 개념 데이터 모델링
- 현실의 엔터티, 속성, 관계를 ERD로 표현.  
- 데이터 설계의 추상 단계.  
- 논리/물리 모델링의 상위 개념.

## 11-5. 스와핑 (Swapping)
- 프로세스 전체를 메모리 ↔ 디스크 교체.

| 단계 | 설명 |
|------|------|
| Swap Out | 메모리에서 디스크로 이동 |
| Swap In | 디스크에서 메모리로 복귀 |

# 📌 12. 분산처리 시스템의 투명성, 스트림 암호

## 12-1. 분산처리 시스템의 투명성

| 종류 | 설명 |
|------|------|
| 접근 투명성 | 로컬/원격 접근 동일 |
| 위치 투명성 | 자원 위치와 무관하게 접근 |
| 이주 투명성 | 자원 이동 시 영향 없음 |
| 복제 투명성 | 여러 복제본이 하나처럼 보임 |
| 병행 투명성 | 다중 접근 시 일관성 유지 |
| 고장 투명성 | 일부 장애에도 서비스 유지 |
| 성능 투명성 | 부하 증가 시 성능 급락 방지 |
| 확장 투명성 | 규모 확장 가능 |

---

## 12-2. 스트림 암호 (Stream Cipher)

- 데이터를 1비트/1바이트 단위로 암호화.  
- 평문 + 키스트림 → XOR → 암호문.
- **키스트림 재사용 금지!**
- 예: RC4(취약), ChaCha20, Salsa20.

# 📌 13. 전체 핵심 요약

## 13-1. 데이터베이스
- 함수적 종속: 결정자 전체에 종속.  
- VIEW: CREATE/REPLACE로 정의, 컬럼명 alias로 변경.  
- DDL/DML/DCL: 구조·조작·권한 제어.  
- 절차형 SQL: 프로시저·함수·트리거.

## 13-2. 운영체제
- 교착상태: 4조건(상호배제·점유대기·비선점·환형대기).  
- 해결방안: 예방·회피·탐지·회복.  
- 디스크 스케줄링: FCFS·SSTF·SCAN·C-SCAN·LOOK.  
- 메모리 관리: First/Best/Worst Fit.  
- 스와핑: 프로세스 전체 교체.

## 13-3. 네트워크/보안
- OSI 전송계층: TCP/UDP.  
- POP3: 메일 다운로드.  
- IPS: 탐지+차단.  
- 인증: 소유/지식/생체 기반.  
- PSK: 사전공유키.  
- BaaS: 블록체인 서비스화.  
- 디지털 서명: 무결성·인증·부인방지.  
- 분산투명성: 자원 통합처럼 보이게.  
- 스트림 암호: XOR 방식, 실시간 암호화.

## 13-4. 기타
- NFC: 근거리 통신.  
- MQTT: IoT 메시징.  
- 오버레이: 메모리 일부만 적재.  
- 바이러스: 복제·은폐·파괴.  
- C언어 포인터: 메모리 연속 구조 이해 필수.
